Purpose

Drive the triage-pg greenfield rewrite to successful completion as scoped by the maintainer on 2026-07-02: a teaching-ready local v1 plus production-monitoring readiness, with the cloud profile's Terraform authored now and its live validation gated on AWS account readiness. The plan additionally performs an ADR conformance audit — verifying that the code fulfills all 25 accepted ADRs and that the ADRs themselves still make sense — and pulls in four previously-deferred items: the UI/UX backlog pair, regression metric_config plumbing, the survival C-index path, and real (OIDC) webapp auth.

Problem

The project is in an unusual, good position: 23 of 25 ADRs are implemented and verified, the full suite is green (394 passed), and three real tutorial datasets run E2E. What remains is not one build but a closure problem — a set of gaps that individually look small but collectively separate "works on my laptop" from "done":

  • ADR drift. Status lines lag reality (ADR-0012 still says the write webapp is "in progress"; ADR-0002 says routing is "partial" though ADR-0025 landed it). One documented contradiction exists: docs/cloud-profile-spec.md §3 declares catwalk/storage.py retired, while its own implementation note admits it was not (still imported by cli.py and 4 test files). Nobody has re-read the 25 decisions end-to-end against the code.
  • ADR-0002 is incomplete. The registry lists projects and routes to them, but creating a project doesn't create or migrate its database — there is no triage project create. Teardown (DROP DATABASE) is likewise manual folklore.
  • ADR-0004/0005 code has never touched AWS. CloudAuth and BatchExecution are moto-tested only; there is no Terraform, no runbook, no triage runs status Batch poll (verified: no describe_jobs call exists anywhere).
  • ADR-0006's payoff is unbuilt. Predictions are append-only precisely to enable monitoring — but no drift views, no scheduled forward scoring, no monitoring panel exist.
  • ADR-0010 is two-thirds done. The survival-ready label schema exists; the survival train/eval path (estimators + C-index) does not. Regression metric_config stops at a default inside in_pg_evaluation.py — not selectable from the experiment config.
  • ADR-0024's auth is a placeholder. TrustedHeaderAuth is laptop-only by design; any shared deployment needs the promised IdP-backed drop-in.
  • Known UX debt. The submit form is a raw-JSON textarea; switching projects on a deep-link 404s (both recorded in TODO.org).

Solution

Seven phases, ordered so that verification precedes construction and every phase leaves main green. Phase 1 audits all 25 ADRs against the code and fixes the decision record itself. Phase 2 closes the local v1 (project lifecycle, storage retirement, the two UX fixes, submit-form validation). Phase 3 completes the problem-type spine (regression metrics, survival C-index — each behind its own config, with a new ADR for survival's implementation choices). Phase 4 lands OIDC auth behind the existing AuthBackend seam. Phase 5 authors the full Terraform + runbook + Batch status backfill, with terraform apply and the live E2E explicitly gated. Phase 6 builds the monitoring layer over the append-only predictions table (scheduled forward scoring + drift SQL views + a dashboard panel, with a new architecture ADR). Phase 7 is the release sweep. Three new ADRs (0026 survival, 0027 monitoring, 0028 OIDC flow) capture the hard-to-reverse choices this plan introduces.

Questionables

Scope questions were answered by the maintainer on 2026-07-02 (target = teaching-ready local v1 + monitoring-ready, with the ADR audit; Terraform authored, apply gated; all four extras in). These are the remaining open decisions — each has a recommended default the build will follow unless overridden.

Survival estimator library — scikit-survival (recommended) vs lifelines

Recommendation: scikit-survival as the optional triage[survival] extra. Its estimators are sklearn-compatible (fit(X, y) with a structured array), so adapters/model.py's existing catwalk seam barely changes; lifelines is DataFrame-first and would re-introduce pandas at the fit seam the project just narrowed. Decided formally in ADR-0026; reversal cost is contained to model.py's estimator mapping.

C-index locus — PL/pgSQL (recommended) vs Python at scoring time

Recommendation: PL/pgSQL, for ADR-0007 consistency (all evaluation in PG) — Harrell's C over comparable pairs with the deterministic tie policy. Risk: naive pair enumeration is O(n²); acceptable at tutorial/consulting eval sizes (≤10⁵ predictions/split ⇒ manageable with an indexed self-join), and ADR-0026 records the Python escape hatch if a client-scale cohort blows past it. The reference cross-check against sksurv.metrics.concordance_index_censored is mandatory either way.

OIDC in-app (recommended) vs oauth2-proxy in front

Recommendation: in-app authlib flow — self-contained, IdP-agnostic via discovery, and testable in the suite; the proxy pattern remains documented as the zero-code alternative since TrustedHeaderAuth behind a trusted proxy already works today. If the deployment reality is "always behind nginx anyway", flipping to the proxy pattern later costs nothing (the seam is one env var).

RDS instance vs Aurora Serverless v2

Recommendation: plain RDS PostgreSQL, smallest viable instance. ADR-0004 names "RDS/Aurora" without choosing; at teaching/consulting scale a db.t4g instance is dramatically cheaper than Aurora's floor, and nothing in the schema needs Aurora. Terraform keeps the engine choice as a variable so the swap is a plan-time decision, not a rewrite.

Webapp project creation — registry-row-only (recommended) vs full DB provisioning

Recommendation: the webapp keeps creating registry rows only; triage project create (CLI, with maintenance credentials) does the CREATE DATABASE + migrate. Rationale: the webapp deliberately holds least-privilege credentials (ADR-0002/0025 kept the registry credential-free); giving the app CREATEDB power would widen its blast radius for a rare operation. The database_ready flag keeps the UX honest about the two-step reality.

When to tag v1.0.0 — after gated cloud E2E (recommended) vs at local completion

Recommendation: v1.0.0-rc1 when Phases 1–4 + 6 + 7 are green; promote to v1.0.0 only after the gated Phase-5 live validation passes, since ADR-0003 declares cloud an always-additive profile but the release claim "two profiles" should not ship with one of them never exercised. If the gate stays closed indefinitely, the maintainer can consciously ship v1.0.0 with a documented "cloud: moto-verified" caveat.

Monitoring reference window — pinned first-quarter (default) vs rolling

Default: the reference window is pinned per model group at deployment (the first stable scoring window), stored so drift is always "vs what we validated". A rolling reference hides slow drift. ADR-0027 records this; the views take the window as a parameter so either policy is queryable.
