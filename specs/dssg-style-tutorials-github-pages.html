Purpose

Give triage-pg what made DSSG triage learnable: a narrative, case-study tutorial per dataset — the full DirtyDuck walkthrough, its 10-minute Dirty Duckling sibling, and new Chicago 311 and DonorsChoose case studies — with the same discussion arc as dssg.github.io/triage's Dirty Duck Tutorial (case framing → infrastructure → data → problem formulation → features → run → evaluate), published on the already-live ccd-ia.github.io/triage-pg Pages site.

Problem

The three tutorial datasets ship as runnable docker stacks with ~100-line runbook READMEs — commands, not teaching. Someone who wants to understand temporal ML on policy data (why point-in-time correctness, why the cohort/label split, what an as-of join buys, when early-warning differs from inspections prioritization) has nothing between the quickstart and the ADRs. DSSG triage solved exactly this with the Dirty Duck tutorial: a narrated case study a student can follow end to end. triage-pg has the datasets, the live Pages site, and (as of migration 0019) the task_framing vocabulary the DSSG tutorial teaches — but no tutorials. Meanwhile the publishing pipeline only knows how to copy two hand-written HTML artifacts; there is no way to author long-form docs comfortably (markdown) and still get styled pages on the site.

Solution

Two layers. (1) An Astro Starlight docs site as the front door (maintainer decision 2026-07-12, over Quarto and mkdocs-material): tutorials are plain markdown under docs-site/src/content/docs/, and Starlight provides the sidebar, client-side search (Pagefind), dark/light themes, and first-class i18n (English now; es/ can land per-tutorial later) — riding the npm toolchain the dashboard already put in CI. The site root becomes the docs index; onboarding.html and the DSSG comparison are passed through public/ so their existing URLs (README, Release assets) keep working. A links-validator plugin fails the build on broken internal links. (2) Four tutorials sharing one chapter skeleton lifted from DSSG's nav (Welcome → Dirty Duckling → Problem description → EIS → Inspections → deeper-look → Infrastructure/Data), adapted per dataset and extended with the surfaces DSSG never had: the dashboard, in-PG audition/fairness, subset evaluations, monitoring, survival, and the derivation DAG. Dirty Duckling is reframed per the maintainer: an installation smoke test (pass/fail, expected outputs quoted) plus a 5-minute dashboard mini-tour.

Questionables

SETTLED 2026-07-12 — generator: Starlight (over Quarto and mkdocs-material)

Maintainer narrowed to Starlight vs Quarto; Starlight chosen. The decider: our tutorial blocks are operational shell+SQL against dockerized databases — Quarto's killer feature (render-time execution proving outputs real) would require the docker stacks inside the docs CI build, so in practice it degrades to freeze-mode local renders, roughly the manual discipline this plan already mandates. Starlight wins the rest for this repo: npm is already first-class in CI (the dashboard), the docs-site-as-index IA makes nav/search/aesthetics the product, and first-class i18n keeps "Spanish later" cheap. Quarto remains the right tool if we ever author executed-notebook analytical tutorials; mkdocs-material was the boring fallback (Python-native, DSSG-familiar) and lost on IA ambition + i18n.

Is "Dirty Duckling" ours to use?

It's an homage, not a claim: DSSG's tutorial is MIT-licensed and triage-pg's README already carries the acknowledgment section crediting DSaPP/CMU for the ideas. Each tutorial's closing chapter cites the original Dirty Duck tutorial explicitly. The alternative (renaming) loses the recognition value for exactly the audience the tutorials target — people coming from DSSG triage.

Does the new inspections config pollute the tutorial DB?

It adds a second experiment to the food DB (different hash — the problem genuinely differs). That's the point: the dashboard's experiment rail showing EIS and Inspections side by side IS the two-framings lesson. The DB is tutorial data; the wipe/re-run path (2026-07-10) exists if it ever needs resetting.

Verbatim-runnability: manual passes or CI automation?
