Purpose

Give triage-pg what made DSSG triage learnable: a narrative, case-study tutorial per dataset — the full DirtyDuck walkthrough, its 10-minute Dirty Duckling sibling, and new Chicago 311 and DonorsChoose case studies — with the same discussion arc as dssg.github.io/triage's Dirty Duck Tutorial (case framing → infrastructure → data → problem formulation → features → run → evaluate), published on the already-live ccd-ia.github.io/triage-pg Pages site.

Problem

The three tutorial datasets ship as runnable docker stacks with ~100-line runbook READMEs — commands, not teaching. Someone who wants to understand temporal ML on policy data (why point-in-time correctness, why the cohort/label split, what an as-of join buys, when early-warning differs from inspections prioritization) has nothing between the quickstart and the ADRs. DSSG triage solved exactly this with the Dirty Duck tutorial: a narrated case study a student can follow end to end. triage-pg has the datasets, the live Pages site, and (as of migration 0019) the task_framing vocabulary the DSSG tutorial teaches — but no tutorials. Meanwhile the publishing pipeline only knows how to copy two hand-written HTML artifacts; there is no way to author long-form docs comfortably (markdown) and still get styled pages on the site.

Solution

Two layers. (1) A minimal markdown→HTML site step: tutorials are authored as plain markdown under docs/tutorials/ (readable and reviewable on GitHub, per the repo's docs philosophy), and a small committed builder (scripts/build_docs_site.py, python-markdown, ~150 lines) renders them into self-contained HTML with the onboarding.html visual identity, a per-page table of contents, and a shared nav strip; docs-pages.yml gains one build step and just docs-site previews locally. Deliberately not mkdocs — that tool was retired with the inherited workflows, and a 150-line renderer is inspectable and dependency-light. (2) Four tutorials sharing one chapter skeleton lifted from DSSG's nav (Welcome → Dirty Duckling → Problem description → EIS → Inspections → deeper-look → Infrastructure/Data), adapted per dataset and extended with the surfaces DSSG never had: the dashboard, in-PG audition/fairness, subset evaluations, monitoring, survival, and the derivation DAG.

[] Phase 1: Site Pipeline — markdown in, styled Pages out

The one-time plumbing every tutorial rides on. Markdown sources stay first-class on GitHub; the builder produces self-contained HTML matching the onboarding identity. No mkdocs, no npm, no external theme — one inspectable script.

1.1. The builder

  • [] uv add --group docs markdown (python-markdown only; extensions: fenced_code, tables, toc, attr_list) — a docs dependency group, not a runtime dep
  • [] scripts/build_docs_site.py: render docs/tutorials/*.md → _site/tutorials/*.html with an embedded house template (teal identity, nav strip linking index · duckling · dirtyduck · chi311 · donors · onboarding · comparison, per-page right-rail TOC from the toc extension); copy docs/onboarding.html → _site/index.html, the comparison page, docs/images/ and docs/tutorials/images/
  • [] --check mode: every relative link resolves inside _site/, every internal anchor exists, every fenced block carries a language tag; exit non-zero on any failure (CI gate)
  • [] just docs-site recipe: build into a temp dir and open the index (local preview without CI)

1.2. Publish plumbing

  • [] docs-pages.yml: replace the hand-rolled mkdir/cp assembly with uv run python scripts/build_docs_site.py --out _site (+ astral-sh/setup-uv); extend the push-trigger paths with docs/tutorials/** and scripts/build_docs_site.py
  • [] docs/onboarding.html: add a Tutorials section (4 cards linking the Pages URLs) so the site index advertises them

1.3. Testing Strategy

The builder is pure file-in/file-out: test it by building. Edge cases: a md file with a broken relative link, a fence without a language, a missing image ref — all must fail --check loudly, not publish broken.

  • [] uv run python scripts/build_docs_site.py --out /tmp/site && uv run python scripts/build_docs_site.py --check — builds clean with a placeholder index.md; check passes
  • [] Feed it a deliberately broken fixture page (bad link + bare fence) — --check exits non-zero naming both
  • [] uv run python -c "import yaml; yaml.safe_load(open('.github/workflows/docs-pages.yml'))" — workflow still parses
🔁 Do not exit this phase until every box above is checked. If any command fails, fix the cause and re-run — loop until all pass.

Questionables

Why a 150-line builder script instead of mkdocs-material (what DSSG actually uses)?

mkdocs was deliberately retired with the inherited workflows (P10: build-mkdocs.yaml deleted, no mkdocs.yml exists), and the repo's docs philosophy is "plain markdown, rendered on GitHub". A pinned python-markdown + an inspectable template keeps that contract: sources readable in-repo, zero theme/config surface, one dependency in a docs-only group. Cost: no built-in search and hand-rolled nav — acceptable for a six-page site. If the site grows past ~15 pages, revisit mkdocs as its own decision (candidate ADR then, not now).

Is "Dirty Duckling" ours to use?

It's an homage, not a claim: DSSG's tutorial is MIT-licensed and triage-pg's README already carries the acknowledgment section crediting DSaPP/CMU for the ideas. Each tutorial's closing chapter cites the original Dirty Duck tutorial explicitly. The alternative (renaming) loses the recognition value for exactly the audience the tutorials target — people coming from DSSG triage.

Does the new inspections config pollute the tutorial DB?

It adds a second experiment to the food DB (different hash — the problem genuinely differs). That's the point: the dashboard's experiment rail showing EIS and Inspections side by side IS the two-framings lesson. The DB is tutorial data; the wipe/re-run path (2026-07-10) exists if it ever needs resetting.

Verbatim-runnability: manual passes or CI automation?

Manual fresh-stack passes per phase (as P8 did for the READMEs), NOT a CI job running dockerized tutorials — that would double CI time and couple docs publishing to docker stacks. The --check gate covers what CI can cheaply prove (links, anchors, fences). Command drift is caught at the next docs touch; if that proves too loose, a nightly tutorial-smoke workflow is the follow-up, out of scope here.

What of DSSG's remaining chapters — ml_governance, aws_batch, choose_your_own_adventure?

Deferred deliberately: governance maps to the model-card/derivation surfaces already covered inside each tutorial; an AWS Batch chapter is gated on the cloud gate opening (it would document unverified steps); choose-your-own-adventure is the tutorials index table in P6. None block the four case studies.

Working notes

  • Dependency: one new dev-group dep (markdown) via uv add --group docs markdown; CI's docs-pages job gains setup-uv but the test job is untouched.
  • Ports discipline: tutorials quote the compose defaults (dirtyduck 5440 · chi311 5438 · donors 5441) — the 2026-07-11 port-drift audit is the cautionary tale; the --check gate can grep for stale port literals as a cheap guard.
  • Voice: the DSSG tutorial's register (narrative, opinionated, "we choose X because Y") is the model; the runbook READMEs stay terse. No content is duplicated between README and tutorial — READMEs run, tutorials teach.
  • Site identity: the builder's template lifts onboarding.html's palette/typography verbatim so the site reads as one artifact; the nav strip is shared by every generated page.
  • Estimated weight: P1 ≈ half a day; P3 is the heavy one (two framings, new config, longest prose) ≈ 1–1.5 days; P2/P4/P5 ≈ half–1 day each; P6 ≈ half a day. The phases are independent after P1, so P3–P5 can interleave with other work or parallelize (plan-team candidate if wanted).
