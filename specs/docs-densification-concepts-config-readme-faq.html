Purpose

Close the conceptual-depth gap between triage-pg's published docs (ccd-ia.github.io/triage-pg) and featurizer's (ccd-ia.github.io/featurizer) by adding four docs surfaces — a Concepts section, a configuration reference, a richer README, and an FAQ — authored as pages on the existing Astro Starlight site and sourced entirely from durable artifacts already in the repo (ADRs, CONTEXT.md, docs/derivation-dag.md, the config validator, and the CLAUDE.md "Common Gotchas"). The bar is depth, not page count: triage-pg's tutorial-first shape is correct for an application and stays; what's missing is the pedagogical "why" and the contracts, which today live only in engineer-facing ADRs and scattered notes.

Problem

Both projects use the same tooling (Astro Starlight, same maintainer), so the density difference is not a stack advantage. It is structural + a real gap:

  • Structural (legitimate): featurizer is a library with a big, stable API surface — 150 feature primitives, a config schema, extensibility registries, output-format contracts. Libraries are reference-dense by nature; its README is ~2× triage-pg's (26 KB vs 14 KB) and ~60% conceptual. triage-pg is an application/pipeline whose value is the end-to-end flow via CLI + dashboard — correctly tutorial-shaped.
  • Real gap (to close): featurizer surfaces a concepts/phi-theory page, an engineering/internals page, an FAQ, and a full config-schema reference. triage-pg's deep "why" — point-in-time correctness, the derivation-DAG identity, the fit-free/fit-based imputation boundary, the ranking spine + problem_type, append-only predictions — lives only in the ADRs (decision records for engineers, not learner pedagogy) and in CLAUDE.md gotchas. The published site has strong tutorials + thin reference, but no concepts, no config reference, no FAQ.

Solution

Four self-contained docs deliverables, each a phase, all landing on the existing docs-site/ (Astro Starlight) and sourced from in-repo durable artifacts so no new "truth" is invented — the plan is transcription + synthesis, not design:

  1. Concepts section (the anchor) — a concepts/ group of focused pages turning the ADRs' short "why" into learner-facing pedagogy.
  2. Configuration reference — one authoritative experiment.yaml page keyed to the validator's _KNOWN_TOP_LEVEL_KEYS, with a drift check.
  3. Richer README — inline the load-bearing contracts + link the Concepts pages, mirroring featurizer's contract-forward README shape.
  4. FAQ — cheap, high-ROI Q/A built from real bug classes and gotchas (including the editable-install fix shipped this session).

English-first; the Spanish /es/ mirror is handled per a decision in Questionables. Navigation is updated in astro.config.mjs; the whole thing is validated by the site build + its links-validator (the same gate docs-pages.yml already enforces).

Questionables

Concepts as Starlight site pages, or docs/metodologia_*.org?

Recommend: site pages. The gap being closed is the published site (that's what's compared to featurizer's phi-theory page). The metodologia skill (→ docs/metodologia_*.org, per CLAUDE.md) is the "long pedagogical why" for partners/students as a PDF — a valuable complementary artifact that can be seeded from the Concepts prose later, but it is a different surface and does not make the site denser. Site pages now; metodologia in Notes as future work.

Spanish /es/ parity now, or English-first?

Recommend: English-first, defer es. The site's es mirror carries a byte-identical-code-block contract (verified last session) that roughly doubles authoring cost. Starlight renders the default-locale page for untranslated routes, so partial coverage does not 404 (verify this behavior in the build during Phase 1 — if fallback is off, either enable it or accept es as a fast-follow). Land en, then mirror es in a dedicated follow-up (same 4-agent pattern used for the existing 10 pages).

Concepts: one dense page (like phi-theory) or several focused pages?

Recommend: 4 focused pages + an index. featurizer has essentially one big idea (DFS/φ). triage-pg has several genuinely distinct concepts (point-in-time correctness, identity/caching, the ranking spine, the data model) that map cleanly to separate ADRs and read better split — matching the existing multi-page reference/ pattern. Keep it to ~4 to avoid sprawl.

Config reference: hand-authored or generated from the validator?

Recommend: hand-authored now, guarded by a drift check. A generated schema page (from a pydantic/JSON-schema model) is more robust but the config is currently validated procedurally in run.py, not via a single declarative schema — generating it is its own project. Hand-author with the _KNOWN_TOP_LEVEL_KEYS drift test so it can't silently rot; note full generation as future work.

Notes

Anti-goal — do not chase page count. featurizer's density is largely earned: it's a consumed library with a 150-primitive API. triage-pg is an application; its tutorial-first shape is correct. Success here is conceptual depth + one config reference + an FAQ, not matching featurizer surface-for-surface. Resist adding a notebooks section or an internals page unless a real need appears.
