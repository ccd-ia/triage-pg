Purpose

Close every remaining functional gap against original DSSG triage (fairness metric completeness + guidance, subset-filtered evaluations, audition single-surface), light up the built-but-dark postmodeling diagnostics and add the missing ones (crosstabs, error analysis, list comparison, calibration), sharpen the model-vs-model-group contrast in both the dashboard and a new CLI inspection surface — then overhaul the README/docs/tutorials with screenshots and diagrams of the final UI, ship two self-contained HTML artifacts (an onboarding one-pager and a triage-pg vs DSSG-triage comparison), stand up a real CI/CD stage on GitHub Actions (the three inherited workflows are stale — one of them attempts a PyPI publish on every tag push and must be retired before rc1), run the full battery, and only then git push, watch CI go green, and tag v1.0.0-rc1. Maintainer decision (2026-07-05): everything before the tag.

Problem

The v1-completion series (11 commits at acb8b668) is functionally complete but not releasable as-is, for four reasons. (1) Documentation lies about the present: the root README still says "Status: early … first runnable slice under construction", there is not a single screenshot anywhere (docs/images/ doesn't exist), and DirtyDuck — the canonical tutorial — has no README at all. (2) Built surfaces are dark: monitoring_calibration() (migration 0012), evaluations_windowed (0010), and the individual_importances table exist with zero routes, CLI, or UI consumers; the crosstabs/error-analysis SQL views promised by ADR-0011 were never implemented. (3) Known parity gaps vs DSSG triage: the bias function stops at {selection_rate, precision, tpr, fpr, fdr} (no FNR/FOR/NPV — and the frontend lists columns the SQL never emits), protected_groups has no ingestion path (test-only inserts), fairness thresholds are hardcoded in React, and subset evaluation is a recorded subset_hash with no actual filtering. The audition dual surface (inherited Python module + SQL views) is the one open conformance flag. (4) No headless inspection: the CLI has zero commands that print a leaderboard, models table, or evaluations — contradicting the headless-complete ethos of ADR-0012.

Solution

Eleven phases in strict order, build first, document last — so every screenshot and both HTML artifacts capture the final UI, and the comparison artifact can say "parity closed" instead of "planned". Phases 1–3 close the DSSG parity gaps in the triage-pg way (PL/pgSQL migrations 0013–0016, config blocks templated like cohort/labels, no Aequitas/pandas dependency). Phases 4–6 deliver postmodeling: first light up the dark SQL, then the two feature-value diagnostics (crosstabs, error trees) via a CLI-computes-PG-persists-dashboard-reads pattern (exactly the ADR-0011 shape), then the model-vs-group contrast surfaces and the headless CLI inspection commands. Phases 7–9 are the documentation lane: README/docs/tutorials rewrite, a scripted Playwright screenshot pass against the live tutorial stack, and the two self-contained HTML artifacts. Phase 10 stands up CI/CD on GitHub Actions — retiring the three stale inherited workflows (one fires a PyPI publish on any tag) and replacing them with a real ci.yml (tests + lint + frontend + terraform + docker) and a tag-triggered release.yml (GHCR image + GitHub Release). Phase 11 is the gate: full battery + three-dataset E2E smokes + conformance closure, then push → CI green on the remote → tag.

Questionables

Subset ranking semantics: re-rank within the subset (chosen) vs filter the global top-k

Two defensible readings of "precision@100 on district 7": (a) re-rank district 7's entities and take its own top-100 — DSSG-compatible, treats the subset as the population; (b) intersect the global top-100 with district 7 — answers "of the people we'd actually act on, how do we do in district 7". The plan implements (a) for DSSG compatibility and because (b) is already answerable via the bias group-by (num_selected/precision per group). If (b) is also wanted later it's a second function parameter, not a schema change.

Error analysis: diagnostic tree only — not a stacked error-correcting model

The maintainer asked "predict on the errors to learn and improve?". Implemented as the DSSG-style diagnostic: a shallow interpretable tree on the error indicator whose rules tell you where the model fails (feeding feature/cohort work). Using the error model to modify scores (stacking/boosting the primary model) is deliberately out of scope: it changes the model-governance story (what is the model hash of a stacked pair?), risks leakage if fitted on test errors, and belongs — if ever — in grid_config as a first-class estimator, not in postmodeling. Recorded here so it isn't re-litigated silently.

Fairness-tree wizard scope: guidance layer, not a gatekeeper

The wizard highlights the relevant metric group and explains why (with the Aequitas fairness-tree credit + link); it never hides metrics or blocks workflows. Rationale: triage-pg is a teaching tool — the tree's pedagogical value is routing attention, and fairness judgment must stay with the human. The intervention: config key only preselects the highlight + the SQL primary_metric stamp.

Crosstab statistics: mean/std/nonzero-rate + ratio only in v1 — no significance testing

Aequitas/DSSG crosstabs shipped t-stats in some variants. The plan persists descriptive stats only: with n in the thousands and heavy feature multiplicity, per-feature p-values invite over-reading; the |log ratio| ranking serves the "what distinguishes the selected" question. Significance testing can be added as another stat value later without schema change.

Screenshots are real-data, scripted, and committed — repo weight accepted

12 PNGs at 1440×900 ≈ 2–4 MB total in-repo. Accepted: GitHub renders them in the READMEs (the whole point), and scripts/capture_screenshots.py makes the set reproducible after UI changes. If weight ever matters, git-lfs or a docs branch is the escape hatch — not worth it at this size.

Breaking CLI change: triage audition loses the audition_config.yaml flow

The old signature ran the inherited AuditionRunner (plots to a directory from a config file). Greenfield replaces it with the hash-addressed, view-backed command. No release has shipped, the repo is pre-tag, and ADR-0001 severed upstream compatibility — acceptable. The comparison artifact's migration-notes strip documents it for DSSG users.

bias_config / evaluation.subsets are identity-neutral (not part of the ADR-0022 problem hash)

Adding bias auditing or subsets to an experiment must not change its experiment_hash — they observe the problem, they don't define it. Precedent: the evaluation: metric block is already identity-neutral. Consequence (accepted): two runs with different subset definitions share the problem hash; subset evaluations are distinguished by subset_hash on their own rows.

Featurizer access in CI: SSH deploy key over an HTTPS PAT

The featurizer pin is git+ssh://…@v0.4.2 in both pyproject.toml and uv.lock. A read-only deploy key on nanounanue/featurizer (loaded via ssh-agent in CI) keeps that pin byte-identical and is repo-scoped least privilege; switching to HTTPS+PAT would rewrite the pin and the lock, grant account-wide scope, and diverge CI from the local/docker build path (which already uses --ssh default). The docker CI job runs on push-to-main only, so the key is never exposed to fork PRs. If featurizer ever becomes public or published, the pin can move to plain https/PyPI and the secret disappears.

CD scope: GHCR image + GitHub Release — no PyPI, no environment deploys

"CD" for a pre-cloud-gate rc means: every tag reproducibly yields the runnable container and a release page carrying the onboarding/comparison artifacts. PyPI publishing is deliberately deleted (the inherited workflow would have fired at rc1) and deferred until the project decides to be installable; deploying infrastructure stays behind the ADR-0004/0005 gate — a future deploy.yml can hang off the same tag trigger once that gate opens. The optional Pages job only publishes static docs.

Migration count: four new migrations (0013–0016) rather than one omnibus

Keeps each parity area independently revertible and reviewable (audition / bias / subsets / diagnostics), mirroring how 0010–0012 landed. The cost (three extra files) is trivial; the benefit shows up the first time one area needs a follow-up fix.

Questionables

Subset ranking semantics: re-rank within the subset (chosen) vs filter the global top-k

Two defensible readings of "precision@100 on district 7": (a) re-rank district 7's entities and take its own top-100 — DSSG-compatible, treats the subset as the population; (b) intersect the global top-100 with district 7 — answers "of the people we'd actually act on, how do we do in district 7". The plan implements (a) for DSSG compatibility and because (b) is already answerable via the bias group-by (num_selected/precision per group). If (b) is also wanted later it's a second function parameter, not a schema change.

Error analysis: diagnostic tree only — not a stacked error-correcting model

The maintainer asked "predict on the errors to learn and improve?". Implemented as the DSSG-style diagnostic: a shallow interpretable tree on the error indicator whose rules tell you where the model fails (feeding feature/cohort work). Using the error model to modify scores (stacking/boosting the primary model) is deliberately out of scope: it changes the model-governance story (what is the model hash of a stacked pair?), risks leakage if fitted on test errors, and belongs — if ever — in grid_config as a first-class estimator, not in postmodeling. Recorded here so it isn't re-litigated silently.

Fairness-tree wizard scope: guidance layer, not a gatekeeper

The wizard highlights the relevant metric group and explains why (with the Aequitas fairness-tree credit + link); it never hides metrics or blocks workflows. Rationale: triage-pg is a teaching tool — the tree's pedagogical value is routing attention, and fairness judgment must stay with the human. The intervention: config key only preselects the highlight + the SQL primary_metric stamp.

Crosstab statistics: mean/std/nonzero-rate + ratio only in v1 — no significance testing

Aequitas/DSSG crosstabs shipped t-stats in some variants. The plan persists descriptive stats only: with n in the thousands and heavy feature multiplicity, per-feature p-values invite over-reading; the |log ratio| ranking serves the "what distinguishes the selected" question. Significance testing can be added as another stat value later without schema change.

Screenshots are real-data, scripted, and committed — repo weight accepted

12 PNGs at 1440×900 ≈ 2–4 MB total in-repo. Accepted: GitHub renders them in the READMEs (the whole point), and scripts/capture_screenshots.py makes the set reproducible after UI changes. If weight ever matters, git-lfs or a docs branch is the escape hatch — not worth it at this size.

Breaking CLI change: triage audition loses the audition_config.yaml flow

The old signature ran the inherited AuditionRunner (plots to a directory from a config file). Greenfield replaces it with the hash-addressed, view-backed command. No release has shipped, the repo is pre-tag, and ADR-0001 severed upstream compatibility — acceptable. The comparison artifact's migration-notes strip documents it for DSSG users.

bias_config / evaluation.subsets are identity-neutral (not part of the ADR-0022 problem hash)

Adding bias auditing or subsets to an experiment must not change its experiment_hash — they observe the problem, they don't define it. Precedent: the evaluation: metric block is already identity-neutral. Consequence (accepted): two runs with different subset definitions share the problem hash; subset evaluations are distinguished by subset_hash on their own rows.

Featurizer access in CI: SSH deploy key over an HTTPS PAT

The featurizer pin is git+ssh://…@v0.4.2 in both pyproject.toml and uv.lock. A read-only deploy key on nanounanue/featurizer (loaded via ssh-agent in CI) keeps that pin byte-identical and is repo-scoped least privilege; switching to HTTPS+PAT would rewrite the pin and the lock, grant account-wide scope, and diverge CI from the local/docker build path (which already uses --ssh default). The docker CI job runs on push-to-main only, so the key is never exposed to fork PRs. If featurizer ever becomes public or published, the pin can move to plain https/PyPI and the secret disappears.

CD scope: GHCR image + GitHub Release — no PyPI, no environment deploys

"CD" for a pre-cloud-gate rc means: every tag reproducibly yields the runnable container and a release page carrying the onboarding/comparison artifacts. PyPI publishing is deliberately deleted (the inherited workflow would have fired at rc1) and deferred until the project decides to be installable; deploying infrastructure stays behind the ADR-0004/0005 gate — a future deploy.yml can hang off the same tag trigger once that gate opens. The optional Pages job only publishes static docs.

Migration count: four new migrations (0013–0016) rather than one omnibus

Keeps each parity area independently revertible and reviewable (audition / bias / subsets / diagnostics), mirroring how 0010–0012 landed. The cost (three extra files) is trivial; the benefit shows up the first time one area needs a follow-up fix.

Answers to the maintainer's two embedded questions

Aequitas fairness tree: yes — incorporate it as the guidance layer (P2.3/P2.4): the tree's two live questions (punitive vs assistive intervention; capped top-k vs broad action) map deterministically onto which disparity family matters (FPR/FDR vs FNR/FOR vs selection-rate). That is exactly the judgment a student/consultant needs help with, and it costs one wizard component + one config key + one docs page. The Aequitas plots themselves (disparity treemaps) are largely covered by the existing disparity bars + pass/fail grid; the treemap adds little over the bars at our group cardinalities.

"Predict on the errors": included as the P5 error tree — the interpretable-diagnostic reading (characterize FP/FN regions to guide improvement). The score-modifying reading (stacking) is consciously excluded — see Questionables.

Risks

  • Ranks-function surgery (P3) is the highest-risk item: labeled_ranks was pinned in 0011 after a live double-join bug. Mitigation: the default-path (subset_hash='') byte-identity regression fixture is written first, before the function is touched.
  • Screenshot stack orchestration (P8) is operationally fiddly (3 dockers + registry + dashboard + seeded runs). Mitigation: the capture script asserts data presence per page before shooting, and the shot list is checked into the script (not ad-hoc).
  • Scope weight: this is roughly a week of build. The phase order front-loads the schema-touching work (P1–P3) so the riskiest changes get the longest soak under the rest of the plan's test traffic.
  • CI first-run unknowns (P11): pytest-postgresql must find the runner's postgres binaries (PATH), and the suite has never run on ubuntu runners in its greenfield shape. Mitigation: actionlint pre-push, --timeout=300 + job timeouts against the historical hang mode, and the gate explicitly loops a red run back to the owning phase instead of tagging.

Explicitly out of scope

  • SHAP / deep interpretability (ADR-0011's deferred companion stands); score-modifying error models (Questionables); Aequitas significance testing; fairness-aware audition rules (a v1.1 candidate — noted in TODO.org at P7); PyPI publishing (the inherited workflow is deleted in P10 — revisit post-v1.0.0); GitHub Pages stays optional/gated (the artifacts work from the checkout).
  • Cloud live validation stays gated (ADR-0004/0005) — unchanged by this plan; the tag is rc1 precisely because that gate is still closed.
